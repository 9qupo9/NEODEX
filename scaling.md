# Масштабирование до миллионов соединений: Упираемся только в RAM

Мы решили проблему передачи рыночных данных миллионам пользователей в реальном времени с помощью двух архитектурных решений:

1. **Zero-Traffic Proxy**: Браузеры клиентов забирают тяжелые котировки по WebSockets *напрямую* с серверов ликвидности (например, Binance). Наш сервер не тратит ни байта трафика и ни процента CPU на их ретрансляцию.
2. **Sharded WebSockets Hub**: Для внутренних уведомлений (ордера, балансы) мы написали свой WS-сервер на Go. Соединения делятся на 256 независимых партиций (шардов). При рассылке алгоритм Fan-Out параллельно задействует все ядра процессора без взаимных блокировок.

### Итог: Суть только в объеме RAM

Мы полностью устранили ограничения процессора (CPU-bound). Одно WS-соединение вместе с горутиной требует всего около **10 КБ** памяти.
Теперь поддержание миллионов онлайнов — это просто расчет стоимости оперативной памяти:
* **1 млн** соединений = **~10 ГБ** RAM.
* **10 млн** соединений = **~100 ГБ** RAM.

---
### Термины
* **Zero-Traffic Proxy** — паттерн, при котором клиент скачивает тяжелые данные напрямую от первоисточника, полностью исключая проксирование трафика через наш сервер.
* **Sharded WebSockets Hub** — разделение единого пула подключений на изолированные части (шарды) для параллельной рассылки событий всеми ядрами CPU без блокировок.
