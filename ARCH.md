# Архитектура NEODEX

NEODEX — это криптовалютная биржа (и интерфейс), написанная на **чистом Go** (Golang) без использования тяжелых сторонних фреймворков и библиотек. Проект сочетает в себе классический Backend-for-Frontend (BFF) и микросервисную готовность к HighLoad.

---

## 🏗 Ключевые архитектурные решения

### 1. Zero-Traffic Proxy (Нулевой трафик на котировках)
Самая ресурсоемкая задача любой биржи — это раздача стаканов (Orderbook) и графиков миллионам пользователей в реальном времени.
В NEODEX мы применили паттерн прямого подключения:
- Браузеры клиентов подключаются к потокам данных **напрямую к серверам Binance** (`wss://stream.binance.com`).
- Наш сервер не тратит **ни одного байта интернет-трафика** на трансляцию рыночных данных, перекладывая эту задачу на плечи глобальных серверов Binance.
- В коде JS реализован **Троттлинг (Batching)**: мы запрашиваем стакан с интервалом `@1000ms` вместо базовых `100ms`, что снижает нагрузку на мобильный интернет клиента ровно в 10 раз без потери визуальной плавности.

### 2. Sharded WebSockets Hub (Шардирование в памяти)
Для рассылки **внутренних** системных уведомлений (исполнение ордеров, обновление балансов кошелька) мы написали собственный WebSocket-сервер на чистом `net/http` (по RFC 6455).
- Чтобы обойти ограничения одного ядра CPU, классический `Hub` был заменен на **`ShardedHub`**.
- Весь пул из миллионов подключений делится на **256 независимых партиций (Шардов)**.
- При поступлении нового события главный Хаб использует алгоритм **Fan-Out**, рассылая сообщение в 256 параллельных Go-каналов (`goroutines`). 
- Это позволяет на 100% утилизировать многоядерные процессоры и держать **от 2 до 10 миллионов** открытых WebSocket-соединений на одном сервере, опираясь только на лимиты физической RAM (~10 КБ на коннект).

### 3. Server-Side Generation (SSG / BFF)
Вместо сложного разделения на Frontend (React/Vue) и Backend (API), мы используем Go как генератор UI.
- Все HTML, CSS и JS файлы "вшиты" прямо в бинарник Go.
- Проект компилируется в один-единственный исполняемый файл `dex.exe`. Нет необходимости настраивать Nginx, Webpack или Node.js. Запустил файл — биржа работает.

### 4. Автоматизированный Сканер Листингов (Background Daemon)
Биржа должна быть живой. Мы написали встроенный демон (goroutine), который:
- Каждый час опрашивает API Binance.
- Сравнивает полученный список пар с локальной базой `data/known_symbols.json`.
- При обнаружении новых токенов сохраняет их в `data/new_listings.json` и выводит в интерфейс с ярлыком **"🆕 NEW"**.

## 📂 Структура проекта (Domain-Driven Design)

* **`cmd/main.go`** — Точка входа (Dependency Injection). Запускает Хабы, сканеры и HTTP-сервер.
* **`internal/api/`** — Слой связи с внешним миром.
  * **`http/`** — REST API (ордера, балансы, сканер) и раздача HTML страниц.
  * **`ws/`** — Кастомная реализация WebSocket протокола (Шардированный Хаб и Клиенты).
  * **`tcp/`** — Бинарный TCP протокол для алгоритмических HFT-ботов (задел на будущее).
* **`internal/domain/`** — Бизнес-логика (Ордера, Аккаунты, Торговые пары).
* **`internal/engine/`** — Ядро сведения ордеров (Orderbook Matching Engine).
* **`internal/ui/`** — Фронтенд-компоненты (HTML/CSS/JS), обернутые в Go-функции.
* **`pkg/decimal/`** — Библиотека высокоточной математики на основе `math/big.Rat`, исключающая баги потери точности (IEEE-754) при работе с балансами.

## 🚀 Путь масштабирования (До 100+ Миллионов)

Текущая архитектура выдерживает всё, что позволит оперативная память (RAM) и лимит дескрипторов ОС (`ulimit`). Когда проект перерастет один физический сервер, код легко масштабируется горизонтально:
1. Ставится балансировщик (AWS ALB / Nginx).
2. Запускаются десятки копий `dex.exe`.
3. Поднимается брокер сообщений (**Redis Pub/Sub** или **Apache Kafka**).
4. Сервера начинают общаться между собой сжатым бинарным форматом **Protobuf**, транслируя миллионам клиентов легкий JSON.
